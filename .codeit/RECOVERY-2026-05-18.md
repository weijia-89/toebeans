# Calibration ledger recovery, 2026-05-18

## What happened

Between 2026-05-16 and 2026-05-18, line 8 of `.codeit/calibration.jsonl` existed in a corrupt state. The line was a single 1047-character string that started life as one valid calibration entry (Entry A: the deprecation-warnings sweep dated 2026-05-16, tier `vibe-safe`, score 92) and ended up with a 68-character fragment of a second entry (Fragment B) glued onto its tail without a separating newline.

Reproducing the structural shape of the corrupted line, eliding the middle:

```
{"date":"2026-05-16","change":"chore: sweep non-blocking deprecation warnings ...","tier":"vibe-safe","score":92,"components":{...},"reviewer":"Wei Jia","notes":"All four migrations verified mechanical: ... compileDebugKotlin now emits zero warnings{"date":"2026-05-16","change":"chore: sweg/jvmTest/assembleDebug)."}
```

Entry A's `notes` field was truncated mid-sentence at the literal `compileDebugKotlin now emits zero warnings`, with no closing quote on the notes string and no closing brace on the parent object. Fragment B was then jammed onto the tail. Fragment B itself was a colliding-truncation artifact: its `change` field read `chore: sweg/jvmTest/assembleDebug).`, where `"swe" + "g/jvmTest/assembleDebug)"` reads like two truncations meeting (the start of `sweep` and the tail of a phrase like `(...debug/jvmTest/assembleDebug)`).

## Likely cause

The 2026-05-16 calibration entries on this line were appended via the `cat >> file <<JSONL` heredoc pattern, which was later identified as a safe-terminal violation. A heredoc terminator failing mid-write produces exactly this shape: the first write's content is interrupted, the second write begins without a fresh newline, and both end up partially captured on a single physical line. This hypothesis is supported by the 2026-05-17 calibration entries on lines further down in the file that explicitly note "this calibration append uses write_to_file + single-line cat, NOT heredoc; previous violations in slices 1 + 2 acknowledged separately."

## Why this matters

The brief's file-wide JSONL validation step (every line must parse as JSON) failed on line 8 from 2026-05-16 until 2026-05-18. Two later vibe-dangerous agents flagged line 8 as a known pre-existing corruption in their own calibration entries and explicitly chose not to repair it, citing the evidence-integrity principle that an agent should not rewrite calibration evidence it did not author.

The 2026-05-18 morning-batch handoff queued the repair as Phase A3, deferred to a separate session under direct user direction.

## Recovery posture chosen

**Option 1: reconstruct Entry A, drop Fragment B.**

Picked from a four-option menu offered to Wei in the 2026-05-18 afternoon orchestration session. The other options were:

- Option 2 (preserve verbatim, exclude line 8 from validation via a sidecar skip file).
- Option 3 (move the corrupt line to a quarantine file, leave a placeholder JSON record at line 8).
- Option 4 (reconstruct Entry A + quarantine Fragment B verbatim).

Wei picked Option 1 with the constraint that the reconstruction must not invent missing content.

## What the reconstruction actually does

The reconstruction script `/tmp/toebeans_fix_line8.py` (run-once script, not committed) preserves every legible byte of Entry A up through the last unambiguous character (`compileDebugKotlin now emits zero warnings`) and then closes the JSON cleanly by appending a single inline truncation marker to the notes field followed by the missing closing quote and brace:

```
... compileDebugKotlin now emits zero warnings [truncated 2026-05-18; see RECOVERY-2026-05-18.md]"}
```

Verbatim chars preserved from Entry A: 979 (the prefix up to and including "zero warnings").
Verbatim chars dropped (Fragment B + collision artifact): 68.
Verbatim chars added (inline marker): 53.
Final line length: 1032 chars.

The inline truncation marker is metadata about the recovery, not content about what Entry A said next. A reader scanning `calibration.jsonl` alone (without consulting this recovery doc) sees the truncation at a glance.

## What is unrecoverable

The original tail of Entry A's `notes` field. Best guess from inspection of neighboring 2026-05-16 entries is a phrase along the lines of "; full gate green" plus a list of Gradle tasks (`ktlintCheck/detekt/testDebug/jvmTest/assembleDebug` or a similar subset). This guess is **not** captured in the reconstructed line because the script's contract was to preserve only legible Entry A bytes.

Fragment B is unrecoverable. The `change` field of Fragment B (`chore: sweg/jvmTest/assembleDebug).`) is a colliding-truncation artifact with no recoverable signal beyond what is preserved verbatim in this recovery doc above.

## Verification

After the reconstruction:

- `.codeit/calibration.jsonl` is 41 lines.
- Every non-empty line parses as JSON.
- Line 8 parses with the expected fields: `date`, `change`, `tier`, `score`, `components`, `reviewer`, `notes`.
- Line 8's `notes` field ends with the truncation marker (verified inline by the reconstruction script).

The 2026-05-18 D1 calibration entry (PR #18, branch `test/d1-argon2id-failing-tests`) is queued to append at line 42 when that PR merges.
