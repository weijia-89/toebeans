## 1. UX warning for confusing dose times — DONE (B9, 2026-05-17)

**Source:** Test-as-spec review 2026-05-15, decision D2 (`docs/adr/0004-tapering-schedule-model.md`).
**Milestone:** 1 (during phase-creation UI).
**What:** When the user enters a `doseTimesLocal` value that falls in `[00:00, 06:00)`, the phase-create screen surfaces a non-blocking warning ("This dose fires between midnight and 6am. Confirm you want to be woken up.") with an explicit "Yes, that's intentional" affirmation.
**Acceptance:** Compose snapshot test + manual smoke test. No automated test for the affirmation copy.
**Why not v0.1:** No UI exists yet; this is layered on top of the phase-create screen which is itself milestone 1 scope.

**Status (2026-05-17):** Shipped in B9. `PhaseDraft.nightDoseWarning` is recomputed on every `updatePhase` call by scanning `doseTimes` for entries in `[00:00, 06:00)`. `ScheduleCreateViewModel.affirmNightDose(index)` is the explicit "Yes, that's intentional" action. Any subsequent edit to dose times resets `nightDoseAffirmed` to `false` so the user re-confirms after changes — the spec was silent on persistence so we defaulted to the safer policy. Banner uses `clearAndSetSemantics` + `LiveRegionMode.Polite` for TalkBack, matching the B8 formError pattern. 5 ViewModel-level tests in `ScheduleCreateNightDoseTest` (boundary cases at 00:00 / 03:00 / 06:00, affirmation clearing, edit-resets-affirmation). Compose snapshot test deferred — the project does not yet use a snapshot framework; the manual smoke test is the v0.1 substitute. Calibration entry in `.codeit/calibration.jsonl`.

## 2. Timezone onboarding + travel-mode toggle

**Source:** Test-as-spec review 2026-05-15, decision D6. ADR-0007 (Accepted with G1-G3 downgrade gates).
**Milestone:** 1.5 (after the phase-create UI exists; before "show schedule for next 30 days" lands).
**What:** Three things:
  - On first pet creation: onboarding screen explaining "your pet's reminders follow your phone's timezone. If you travel without your pet, mark that on the pet's profile."
  - On `Pet` entity: optional `homeTimezone` field. When set, materialization uses this TZ instead of `TimeZone.currentSystemDefault()`.
  - On `Schedule` entity: optional `travelMode` enum (`FOLLOW_PHONE` | `STAY_HOME_TZ` | `ELAPSED_INTERVAL`). Default `FOLLOW_PHONE`.
**Acceptance:** Full G2 closure — `SchedulePhaseDstRulesTest` (still NOT YET CREATED, see ADR-0007 § Verification status), plus DST_SKIP / DST_DUPLICATE_RESOLVED warning surfaces on `ScheduledDose`, plus integration test simulating `ACTION_TIMEZONE_CHANGED`.
**Why not v0.1:** Requires design work — see ADR-0007 "Open design choices."

**Status (2026-05-17):** ADR-0007 partially advanced. G2 partial: 4 DST-related cases in `SchedulePhaseRulesTest` now pin kotlinx-datetime behavior (the snap-to-nearest-valid-instant default). Full surface still missing. G1 (Plumb's drug-list validation) and G3 (n=3-5 user research on prompt copy) both still open. See ADR-0007 § Verification status (2026-05-17) for gate-by-gate detail.

## 3. Real DB lookup in `DoseAlarmReceiver`

**Milestone:** 1.
**What:** On `BOOT_COMPLETED`, an entry-point in `ToebeansApp` runs `ScheduleCalculator.computeScheduledDoses` for every active schedule across a 72h window and re-schedules AlarmManager entries.

**Status (2026-05-17):** ADR-0012 (`docs/adr/0012-boot-completed-receiver-lifecycle.md`) specifies the receiver lifecycle. **Implementation NOT YET SHIPPED** — no `BootCompletedReceiver.kt` exists in `androidApp/src/main/kotlin/app/toebeans/android/notifications/`. Receiver implementation depends on the SQLDelight persistence layer (M1 Tier C) because the receiver needs real schedule data to re-materialize against.

## 7. Argon2id key derivation for backup cipher

**Milestone:** 2.
