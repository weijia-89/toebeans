## 1. UX warning for confusing dose times: DONE (B9, 2026-05-17)

**Source:** Test-as-spec review 2026-05-15, decision D2 (`docs/adr/0004-tapering-schedule-model.md`).
**Milestone:** 1 (during phase-creation UI).
**What:** When the user enters a `doseTimesLocal` value that falls in `[00:00, 06:00)`, the phase-create screen surfaces a non-blocking warning ("This dose fires between midnight and 6am. Confirm you want to be woken up.") with an explicit "Yes, that's intentional" affirmation.
**Acceptance:** Compose snapshot test + manual smoke test. No automated test for the affirmation copy.
**Why not v0.1:** No UI exists yet; this is layered on top of the phase-create screen which is itself milestone 1 scope.

**Status (2026-05-17):** Shipped in B9. `PhaseDraft.nightDoseWarning` is recomputed on every `updatePhase` call by scanning `doseTimes` for entries in `[00:00, 06:00)`. `ScheduleCreateViewModel.affirmNightDose(index)` is the explicit "Yes, that's intentional" action. Any subsequent edit to dose times resets `nightDoseAffirmed` to `false` so the user re-confirms after changes. The spec was silent on persistence so we defaulted to the safer policy. Banner uses `clearAndSetSemantics` + `LiveRegionMode.Polite` for TalkBack, matching the B8 formError pattern. 5 ViewModel-level tests in `ScheduleCreateNightDoseTest` (boundary cases at 00:00 / 03:00 / 06:00, affirmation clearing, edit-resets-affirmation). Compose snapshot test deferred. The project does not yet use a snapshot framework; the manual smoke test is the v0.1 substitute. Calibration entry in `.codeit/calibration.jsonl`.

## 9. Midnight-straddle UX warning during phase creation: DONE (2026-05-18)

**Source:** ROADMAP § Milestone 1 "Midnight-mode UX warning during phase creation". Separate from item #1 (which shipped the `[00:00, 06:00)` night-dose / sleep-disruption nudge in B9); this item covers the distinct calendar-day-boundary case where the set of dose times itself wraps around midnight (e.g. one dose at 23:00 and another at 01:00 within the same phase).
**Milestone:** 1.
**What:** When `PhaseDraft.doseTimes` straddles midnight per `MidnightStraddleDetection.crossesMidnight` (the rule: after sorting by second-of-day, some interior gap is strictly larger than the wrap-around gap), `ScheduleCreateViewModel.updatePhase` sets `PhaseDraft.crossesMidnight = true`, and `PhaseEditorCard` renders an info-tinted (`secondaryContainer`) banner above the dose-time editor reading "Heads up: your dose times cross midnight. Doses after midnight land on the next calendar day." Non-blocking; no affirmation action. Co-exists with item #1's `nightDoseWarning` (`tertiaryContainer`): both can fire for a set like `[23:00, 01:00]` because the same configuration triggers both rules. The two banners are visually distinct.
**Acceptance:** 9 unit tests, all green. 7 in `MidnightStraddleDetectionTest` pin the detection rule including the empty / single-dose / single-00:00 / typical-daytime / 23-and-01 / three-dose-spanning / symmetric-tie cases. 2 in `ScheduleCreateMidnightStraddleTest` pin the ViewModel state transitions: editing into a straddling configuration flips the flag on, and editing back out clears it.

**Status (2026-05-18):** Shipped. Banner accessibility matches the formError + nightDoseWarning pattern (`clearAndSetSemantics` + `LiveRegionMode.Polite` + `contentDescription`) so TalkBack announces once on appearance. The detection function is pure and lives next to the form state (`androidApp/.../ui/schedule/MidnightStraddleDetection.kt`) rather than in the scheduler proper: it is purely a UX-derivation flag, not a domain property, and is not persisted on `SchedulePhase`. Bedtime + morning cases like `[23:00, 07:00]` do straddle by the mechanical rule (16h day-gap > 8h night-gap); the strict-inequality wrap-vs-interior rule was chosen for its mechanical clarity, and the UX team can revisit whether to soften it (e.g. minimum wrap-gap floor) later if real users report false positives.
